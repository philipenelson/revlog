# Session: Real session restoration ‚Äî POST /auth/refresh, token rotation, and silent restore-on-mount

**Date:** 2026-06-08
**Branch/worktree:** `worktree-garage-wire-backend`

---

## Goal

User-reported bug: "Once I'm in the garage screen, if I reload the screen, I get sent back to the login screen. If I try to load directly the `/garage` route, it also sends me back to the login screen." This was the *expected* fallout of the immediately-prior session's fix (`2e30de5` ‚Äî redirect to `/login` instead of a dead-end load error on a missing session): that fix made the gap *recoverable* (one extra sign-in) but didn't close it. The actual gap ‚Äî "no session restoration on reload" ‚Äî was named explicitly in [ADR 0016](../adr/0016-client-session-and-route-protection.md)'s Consequences and tracked as "Token rotation on refresh" in [`v1.md`](../milestones/v1.md). This session builds the real fix end to end: ADR + spec, backend endpoint with rotation and tests, client-side wiring, and updated E2E coverage.

## Root cause

`AuthProvider` holds the session in React state only ‚Äî in-memory, never `localStorage` (ADR 0016, the XSS-safe pattern from [ADR 0002](../adr/0002-custom-jwt-auth.md)). A reload or direct navigation wipes that state. The `HttpOnly` `refreshToken` cookie survives ‚Äî it's what gets the visitor past Next.js middleware ‚Äî but until this session there was no mechanism to exchange it for a fresh session. The garage screen correctly detected "no session" and redirected to `/login` (per `2e30de5`), but a still-signed-in user shouldn't have to re-authenticate at all.

## Key decisions

- **Rotation, not reuse**: every successful `POST /auth/refresh` deletes the matched `RefreshToken` row and inserts a new one ‚Äî exactly what [ADR 0012](../adr/0012-refresh-token-storage.md) specified ("rotation deletes the old row and inserts a new one"). Full rationale in the new [ADR 0017](../adr/0017-refresh-token-rotation.md).
- **Single, indistinguishable 401**: "no cookie," "hash matches no row" (garbage, forged, or already-rotated), and "expired" all collapse into one `401 Invalid or expired session` ‚Äî mirrors `login`'s precedent and is the explicit, documented trade-off for deferring full reuse detection (which would require retaining rotated-token provenance ‚Äî a schema change beyond V1 scope; tracked as a V2+ item in ADR 0017).
- **Restoration lives in `AuthProvider`, once, not per-page**: a single `useEffect` on mount calls `POST /auth/refresh` whenever `session` is `null`, and exposes a new `isRestoring: boolean` so consumers can distinguish "still checking" from "confirmed logged out." This is the actual fix ‚Äî the garage screen's existing `if (!session) router.replace("/login")` fired on the very first render, where `session` is always initially `null`, before the silent restore had a chance to populate it. Gating that redirect behind `isRestoring` settling is what stops the "reload sends me back to login" flash-redirect.
- **UC-AUTH-5 bundled in**: `v1.md` explicitly tied "authenticated user visiting `/login` is redirected away" to "Token rotation on refresh" ("also unblocks‚Ä¶"), and once `session` + `isRestoring` exist in `AuthProvider`, wiring the login screen to redirect via `routeForAccountStatus` needed almost no extra plumbing ‚Äî judged in-scope rather than scope creep. Implemented **client-side**, not via Next.js Edge middleware as the original AC assumed ‚Äî middleware can confirm a session cookie is *present* but not which account status it belongs to (ADR 0016) ‚Äî which means the form can be visible for the brief duration of the silent-refresh request before the redirect fires. Gating the form behind a loading state would also delay it for the far more common fresh-visitor case, so that small flash is an accepted, documented trade-off.
- Explicitly did **not** extend redirect/guard logic to the onboarding screen ‚Äî out of scope (not reported, and the silent restore resolves the equivalent dead end there as a side effect in the common case).

## Explicitly out of scope (tracked for later)

- **Full token-reuse detection and mass session revocation** ‚Äî [ADR 0012](../adr/0012-refresh-token-storage.md) names the ideal ("hash not found ‚Üí revoke all sessions for that user"); [ADR 0017](../adr/0017-refresh-token-rotation.md) explains why it's deferred (requires retaining rotated-token provenance ‚Äî a schema change beyond this endpoint's scope) and tracks it as a V2+ item
- **Silent renewal of an access token that expires *mid-session*** (as distinct from restoration on reload/direct navigation, which this session solves) ‚Äî e.g. transparently retrying a 401'd `apiFetch` call after a refresh, or proactively refreshing before the 15-minute TTL elapses. `POST /auth/refresh` now exists and could power this, but nothing calls it reactively or proactively from an active session yet ‚Äî the same dead-end-error pattern `2e30de5` fixed for "no session on load" likely still exists for "session expired mid-use." This was previously conflated with the restoration-on-reload AC; reframed as its own, still-open item in `login.md`
- `POST /auth/logout` ‚Äî named alongside `refresh` in ADR 0012 as completing the session model; a separate endpoint with its own spec
- The brief login-form flash for already-authenticated visitors while the silent restore is in flight ‚Äî an accepted trade-off (see Key decisions); closing it would require a loading gate that slows the far more common fresh-visitor path
