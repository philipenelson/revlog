# Mobile sync: outbox pattern with SyncService

## Context

The mobile app is offline-first: all reads come from the local SQLite database and all writes must be possible without a network connection. When connectivity is restored, local changes must be reliably delivered to the API without duplicates.

Three sync approaches were considered:

**Pull-everything on reconnect** — on reconnect, push any pending local changes then fetch all Vehicles and Log Entries from the API and overwrite the local database. Simple to implement and viable given the small data volumes an Owner accumulates. However, it does not handle the case where a record was deleted: the API returns only what exists, so a locally-deleted record would be restored on the next pull. It also re-fetches the entire dataset on every reconnect, which is wasteful.

**Incremental sync with timestamps** — track an `updatedAt` watermark and pull only records modified since the last sync. Requires soft deletes on the API (hard deletes are invisible to a timestamp query). The current API uses hard deletes on Vehicles with cascading child deletion; converting to soft deletes is significant additional API scope on top of the mobile work.

**Outbox pattern** — every write (create, update, delete) is applied to the local SQLite database immediately, and an outbox entry is written in the same database transaction. A `SyncService` reads the outbox when online and sends each pending mutation to the API in order. The UI always reads from the local database and never waits for the network.

## Decision

Use the **outbox pattern**. Writes go to SQLite and the outbox in a single atomic transaction. The `SyncService` flushes the outbox to the API when online.

### Outbox table schema

```sql
CREATE TABLE outbox (
  id            TEXT PRIMARY KEY,     -- UUID, also the idempotency key
  type          TEXT NOT NULL,        -- e.g. 'CREATE_LOG_ENTRY', 'DELETE_VEHICLE'
  payload       TEXT NOT NULL,        -- JSON string
  created_at    INTEGER NOT NULL,     -- Unix timestamp ms
  status        TEXT NOT NULL DEFAULT 'pending'
                                      -- 'pending' | 'processing' | 'failed'
);

CREATE UNIQUE INDEX outbox_id ON outbox(id);
```

The `id` doubles as an idempotency key. If a mutation is sent to the API and the response is lost before the client receives it, retrying with the same `id` is safe — the server either recognises the key and returns the previous result, or the mutation is truly new. New idempotency-keyed endpoints are added to the API as needed for mutating operations.

### SyncService

`SyncService` lives in `infrastructure/sync/SyncService.ts`. It:

1. Subscribes to network state changes (`@react-native-community/netinfo`)
2. On connectivity restored: flushes the outbox — processes `pending` entries in `created_at` order, one at a time, marking each `processing` then deleting on success or `failed` on unrecoverable error
3. On explicit refresh (pull-to-refresh or app foreground): pulls all data from the API and reconciles the local database (upsert for existing records, delete locally any record not present in the API response)
4. Exposes `pendingCount` — the count of unsynced outbox entries — so the offline indicator can reflect unsent changes even when the device is online

### Conflict resolution

Server wins. If the API rejects a mutation (e.g. the record was deleted server-side since the local write), the outbox entry is marked `failed` and the local record is reverted on the next pull. This is the correct policy for a single-user app: the server is the source of truth, and conflicts only arise from multiple devices or sessions.

### Offline indicator

The header shows the offline indicator when `networkState.isConnected === false` OR `outboxPendingCount > 0`. This tells the Owner that either they are offline, or they are online but local changes have not yet been confirmed by the server.

### Update (2026-07-02): ordered multi-collection reconcile + client-generated ids

Gap found during the mobile Garage screen's design: this ADR describes reconciling "the local database" and flushing "the outbox" as if each were a single collection, but Vehicles and Log Entries are parent/child, and two ordering problems fall out of that which the original decision didn't address:

- **Pull-side**: if Vehicles and Log Entries are reconciled independently with no ordering guarantee, a Log Entry could transiently reconcile before its parent Vehicle exists locally (or after its parent has just been removed), and there is nothing in the original decision that rules this out.
- **Push-side**: a Log Entry created offline for a Vehicle also created offline references a Vehicle the server doesn't know about yet. If Vehicle ids are server-assigned (as `POST /vehicles` does today — no client-supplied `id`), the Log Entry's outbox entry cannot be resolved until the Vehicle's own outbox entry has been confirmed and its real id returned.

Two approaches were considered for the pull-side ordering problem beyond the two already in this ADR's Context (pull-everything, incremental-with-timestamps):

- **Soft delete + timestamp watermark** — this is already this ADR's "Incremental pull" V2+ item. It solves re-fetch cost and delete visibility more efficiently at scale, but does not by itself remove the parent-before-child ordering requirement — an incremental pull still needs to reconcile Vehicles before Log Entries within a batch. Confirmed as the right upgrade path once data volume/query cost justifies the added API scope (schema migration, cascading soft-deletes, every read needing a filter) — not yet, at this app's current scale (an Owner's handful of vehicles).
- **Phased reconcile within the existing hard-delete + full-replace approach** — reconcile parent collections to completion before starting dependent child collections, deriving the child pull's scope (which Vehicle ids to fetch Log Entries for) from the just-reconciled local parent state, not a pre-sync snapshot. No cross-collection transaction is needed: each phase is its own complete, atomic single-collection replace, so an interrupted sync leaves some collections stale, never a dangling reference. Combined with `PRAGMA foreign_keys = ON` and `ON DELETE CASCADE` on child tables (`log_entries.vehicle_id REFERENCES vehicles(id) ON DELETE CASCADE`), a parent removed by its collection's reconcile automatically removes its now-orphaned local children — mirroring the API's own cascading hard deletes (see the delete-vehicle spec) with no application code required.

**Decision:** use phased reconcile + FK cascade (the second option) for the pull side — it requires no API changes and no new decision beyond sequencing `SyncService.pull()`'s phases correctly. For the push side: entities that can be created offline and referenced by other offline-created entities (Vehicle, referenced by Log Entry) get a **client-generated id** (UUID, generated at local-create time and sent as-is to the API's create endpoint) rather than a server-assigned one. Combined with the outbox flush already processing entries in `created_at` order and *stopping* the whole flush (not skipping ahead) on a retryable failure, a dependent entity's mutation can never be sent before its parent's, because it was enqueued after it and the queue halts rather than reordering around a failure. This requires the relevant create endpoints (starting with `POST /vehicles`) to accept a client-supplied `id`, which they do not today — a prerequisite for whichever future work adds offline vehicle/log-entry creation (Add Vehicle, New Log Entry), not something this update implements itself.

This does not change the outbox pattern, the outbox table schema, or the server-wins conflict policy above — it refines *how* multi-collection reconcile is sequenced and *how* dependent offline-created entities get ordered on push. No code in the mobile Garage screen work ships against this yet (it reconciles a single collection, Vehicles, with no children); this is captured now so `LogEntryRepository` and Add Vehicle have a documented answer when they're built, per this repo's rule that a decision isn't made until it's written down.

### Update (2026-07-03): child-collection pull sourced from per-vehicle `GET /vehicles/:vehicleId`, not a standalone list endpoint

Gap found while building the Vehicle Detail screen (`LogEntryRepository`, the first consumer of the 2026-07-02 update above): that update assumed Log Entries would be pulled as a flat child collection, but the API has no endpoint that lists Log Entries across all of an account's Vehicles — the only ways to read a Vehicle's Log Entries are `GET /vehicles/:vehicleId` (full `VehicleDetail`: identity, insurance, `logEntries`, `stats`, `transferPending`/`pendingTransfer`) or `GET /vehicles/:vehicleId/log/:entryId` (one entry). Phase 2 of `pull()` therefore fetches the whole `VehicleDetail`, not just its `logEntries`.

**Decision:** phase 2 iterates the Vehicles just reconciled by phase 1 and calls `getVehicle(client, vehicle.id)` for each (N calls for N Vehicles — accepted at V1 scale for the same reason as this ADR's original "small data volumes" reasoning; revisit if an Owner's Vehicle count grows large enough to matter). For each response:

- The local Vehicle row is extended in place (`VehicleRepository.applyDetail`) with the fields `VehicleSummary` doesn't carry: `stats.totalSpent`, `stats.lastLoggedAt`, `transferPending`, and `pendingTransfer.recipientEmail` — this is what makes the Vehicle Detail screen's stats strip and locked-transfer state renderable from local SQLite alone, with no request in the render path.
- `logEntries` from every response are collected into one array and reconciled **once**, after the per-vehicle loop, via `LogEntryRepository.reconcile()` — a single full-collection replace, consistent with how `VehicleRepository.reconcile()` already works. This keeps `Store<T>.replaceAll()` as the only atomicity primitive needed; no per-vehicle-scoped replace was added to the `Store<T>` port.
- A per-vehicle `getVehicle` failure (network blip, or the Vehicle was deleted server-side between phase 1 and phase 2) is logged and skipped rather than aborting the whole pull — that Vehicle's detail fields and Log Entries simply stay stale until the next sync, which is the same "an interrupted sync leaves some collections stale, never a dangling reference" tradeoff this ADR already accepted for the phased-reconcile design.

**`insurance` is fetched but not persisted.** `VehicleDetail.insurance` comes back on every one of these calls, but no mobile screen reads or writes insurance yet (Vehicle Detail's V1 design has no insurance UI — see `docs/specs/mobile-app/vehicle.md`'s Decisions). Storing it now would be a local column with zero readers, which this repo's guidance is to avoid; it is discarded in `SyncService.pull()` today. This is deliberately deferred, not forgotten — captured here for whichever future work adds mobile insurance display so it has a documented starting point (probably a further extension of `applyDetail`) rather than re-deriving this call site from scratch.

**A failed per-vehicle detail fetch must not regress that Vehicle's already-known data.** Phase 1's `reconcile()` still has to run on every pull (it's the only phase that can add/remove Vehicles), but `VehicleSummary` alone can't populate the detail columns — so `reconcile()` looks up each Vehicle's current local row before replacing the table and carries its existing detail fields forward (falling back to `DEFAULT_DETAIL` only the first time a Vehicle is seen), rather than blanking them and waiting for phase 2 to refill them. Phase 2 then does the same for Log Entries on a failed fetch: it re-reads that Vehicle's current local entries via `LogEntryRepository.findByVehicleId()` and folds them back into the combined list before the single `reconcile()` call, since that call always replaces the whole table. Net effect: a Vehicle whose `getVehicle` call fails this cycle keeps exactly what it had after the last successful sync — stats, `transferPending`, and Log Entries all included — with no dependence on phase ordering within a single `pull()` to avoid a visible regression.

This does not change the outbox pattern, the outbox table schema, the server-wins conflict policy, or the 2026-07-02 update's phased-reconcile/FK-cascade/client-generated-id decisions — it answers the specific question that update left open ("what does `LogEntryRepository` actually pull from") now that `LogEntryRepository` exists.

### Update (2026-07-03): `OutboxWriter<T>` — cross-table atomic write port for offline entity writes

Gap found while building Edit Vehicle (UC-MOB-VEH-3), the first mobile screen that actually writes offline: this ADR's Decision says "Writes go to SQLite and the outbox in a single atomic transaction," but no code path implements that yet. `Store<T>` — the only write primitive that exists — is deliberately scoped to one collection (this ADR's 2026-07-02 update: "no cross-collection transaction() method... each phase is its own complete, atomic single-collection replace"), and a write that also enqueues an outbox entry touches two collections (the entity's table and `outbox`) that don't share a `Store<T>` instance: `DatabaseProvider` constructs one `Store<T>` per table, so a repository's `update()` has no way to reach the outbox table's rows through the store it's given.

Two approaches were considered:

- **Sequential writes** — the repository calls `store.save()` then a separate `outboxRepository.enqueue()`, un-transacted. Simplest, but violates this ADR's own atomicity requirement: a crash or app kill between the two calls leaves a locally "saved" edit that never reaches the outbox, silently losing the write with no user-visible signal — exactly what the outbox pattern exists to prevent.
- **`OutboxWriter<T>`, a second, narrower port** — `save(record: T, outboxType: string, outboxPayload: unknown): Promise<void>` that writes both tables inside one `db.transaction()` call. Kept separate from `Store<T>` rather than folded into it, because `Store<T>` is deliberately generic and SQL-agnostic (its own header comment: "not named or shaped after a database... nothing here presupposes SQL") and used everywhere, including read paths and the pull-side `replaceAll()` that have no outbox involvement at all. Making every `Store<T>` call site outbox-aware would be a much bigger, unrelated change for a capability only offline-write repositories need.

**Decision:** add `OutboxWriter<T>` (`infrastructure/database/OutboxWriter.ts`), implemented by `createOutboxWriter(db, table)` in `SQLiteStore.ts` using the same synchronous `db.transaction()` primitive `SQLiteStore.replaceAll()` already uses — one transaction upserts the entity row (`onConflictDoUpdate`, same semantics as `Store<T>.save()`) and inserts the outbox row, so a crash between the two is impossible: either both commit or neither does. A repository that needs this (today: `VehicleRepository.update()`; the same shape will serve `create`/`delete` and later `LogEntryRepository` writes) is constructed with both its `Store<T>` (for reads and non-outbox writes) and an `OutboxWriter<T>` scoped to the same table. `DatabaseProvider` wires both from the same `DrizzleDb` instance.

This does not change the outbox pattern, the outbox table schema, or the conflict policy — it is the concrete mechanism this ADR's original Decision described but never specified, now that a write path actually needs it.

### Update (2026-07-03): `POST /vehicles` accepts a client-supplied `id`

Fulfils the prerequisite the 2026-07-02 update flagged and deferred: "this requires the relevant create endpoints... to accept a client-supplied `id`, which they do not today... a prerequisite for whichever future work adds offline vehicle/log-entry creation (Add Vehicle, New Log Entry)." Add Vehicle (UC-MOB-VEH-2) is that work.

**Decision:** `createVehicleSchema` (`packages/domain`) gains an optional `id: z.uuid()` field, validated and forwarded end-to-end — `CreateVehicleData`/`IVehicleRepository.create` accept it, and `PrismaVehicleRepository.create` uses `vehicle.upsert({ where: { id }, create: data, update: {} })` instead of a plain `create` whenever an `id` is supplied. The web client never sends one, so `Vehicle.id`'s Prisma default (`@default(uuid())`) still applies there unchanged.

The upsert-with-no-op-update (rather than a plain `create`) is what makes this ADR's original idempotency promise ("retrying with the same id is safe") actually hold for creates specifically: `VehicleRepository.create()` on mobile (see below) generates the id at local-write time, before the outbox entry has been sent even once, so a retried `CREATE_VEHICLE` outbox entry — e.g. the request reached the server and committed, but the response was lost before the client saw it — resolves to the same row instead of a unique-constraint error.

**Known limitation, deliberately not solved here:** this only covers the "same id resubmitted" case at the row level. A retried create is still classified by `outboxHandlers.ts`'s existing `isRetryable()` the same as any other mutation (5xx/network → retryable, 4xx → permanent) — it does not special-case "this looks like the same request landed already." That's fine for the common failure mode (the request never reached the server), and the upsert above means it's also fine for the "reached the server, response lost" case. Revisit only if a distinct failure mode turns up in practice that this doesn't cover.

**Adjacent bug found and fixed while wiring `CREATE_VEHICLE`'s payload:** `createVehicleSchema`'s `nickname` field only accepted `undefined` (the web client's shape — `draft.nickname.trim() || undefined`) or a real string, not an explicit `null`. Every mobile outbox payload (`CreateVehicleData`/`UpdateVehicleData`, both typed `nickname: string | null`) sends the field with a value, never omits the key, so a blank nickname serializes to `"nickname": null` on the wire — which the schema rejected outright (400) before this fix. This would have broken the common "no nickname" case for both `CREATE_VEHICLE` (new, this update) and the already-shipped `UPDATE_VEHICLE` (Edit Vehicle, whenever a nickname is cleared — a path with no existing test coverage, which is how it went unnoticed). Fixed by adding `.nullable()` to the field in `packages/domain/src/schemas/vehicle.ts`, a pure widening: `null`, `undefined`, and a real string all still collapse to the same transformed output. See `docs/specs/mobile-app/vehicle.md`'s Decisions for the spec-level record.

**Mobile side:** `VehicleRepository.create()` generates the id via `Crypto.randomUUID()` (the same primitive `OutboxRepository` and `secureStorage` already use for client-generated ids elsewhere in this app) and writes it through `OutboxWriter<T>` — entity row and `CREATE_VEHICLE` outbox entry in one transaction, the same shape this ADR's 2026-07-03 `OutboxWriter<T>` update already established for `update()`. `outboxHandlers.ts` gets a `CREATE_VEHICLE` handler calling `createVehicle(client, { id, ...data })`, classified retryable/permanent identically to `UPDATE_VEHICLE`'s existing handler.

This does not change the outbox pattern, the outbox table schema, or the conflict policy — it closes the one open prerequisite the 2026-07-02 update left for offline entity creation.

### Update (2026-07-03): offline-durable photo upload on Add Vehicle

Gap found immediately after shipping Add Vehicle (UC-MOB-VEH-2) without it: the mobile spec (`docs/specs/mobile-app/vehicle.md`) and the `revlog-mobile-add-vehicle.html` design file both already said photo upload was deferred to V2, and the first pass took that at face value and shipped a static placeholder matching it. That deferral was never actually confirmed as still wanted before building against it — it wasn't, and this update reverses it. The broader lesson (not solved by this update, a process note): a pre-existing "V2" cut sitting in a spec/design file is not the same as it having been re-confirmed for the work at hand — worth a check-in before quietly building to it, not just before *introducing* a new cut.

The web app's photo upload is possible in one line — pick a `File`, hold it in memory, send it in the same `POST /vehicles` multipart request — because the web app has no offline story; a request either goes out now or the whole submit fails now. Add Vehicle's create, per this ADR, does neither: it writes to SQLite and an outbox entry instantly, and the actual `POST /vehicles` may not fire until long after, possibly after an app restart. A picked photo has to survive that gap, which the web app's approach has nothing to say about — this update is the mobile-specific mechanism.

Two approaches were considered:

- **A second outbox entry type** (e.g. `UPLOAD_VEHICLE_PHOTO`), enqueued alongside `CREATE_VEHICLE`, uploading via the existing `POST /vehicles/:id/photo` endpoint once the Vehicle it references has synced. Correct in principle — the outbox already guarantees strict per-entry ordering, so a photo entry enqueued after its Vehicle's create entry can't jump ahead of it, the same guarantee the 2026-07-02 update relies on for a Log Entry referencing an offline-created Vehicle. But it needs a local schema field to track "photo still pending" so the UI can show *something* in the meantime (and to know when to stop trying), which is a real, ongoing piece of state to keep in sync with `VehicleRepository.reconcile()`'s existing carry-forward logic for every other local-only field.
- **Bundle the photo reference into `CREATE_VEHICLE`'s own payload**, handled by one call that does a plain create or a multipart create-with-photo depending on whether a photo reference is present. No second entry type, no ordering question (there's only one entry), no local "pending" field to track or reconcile — the local row simply doesn't have a photo-related column at all; the next full sync's existing `reconcile()`/`applyDetail()` picks up the confirmed `photoUrl` exactly the way it already does for a Vehicle created without a photo.

**Decision:** bundle it into `CREATE_VEHICLE` (the second option). Mechanism:

1. **Picking**: `expo-image-picker`'s `launchImageLibraryAsync` (library only, no camera capture — matches the web app's plain file input, not a broader capability). Requires the `expo-image-picker` config plugin (photo library usage description) in `app.config.ts`; like `expo-sqlite`'s `useSQLCipher` flag, this needs a native rebuild (`expo prebuild`/dev-client), not available in Expo Go.
2. **Persisting**: before `VehicleRepository.create()` writes anything, if a photo was picked, its (possibly cache-scoped, picker-owned) file is copied via `expo-file-system` into this app's own stable storage — `${FileSystem.documentDirectory}vehicle-photos/<vehicleId>.<ext>` — so it survives the picker's own temp files being cleared and survives an app restart. `<vehicleId>` is the same client-generated id `create()` already produces (2026-07-03's earlier update, above), so the path is derivable without storing anything extra.
3. **Payload**: the `CREATE_VEHICLE` outbox entry's payload gains an optional `photo: { uri, name, type }` — the stable local path plus the filename/MIME type `createVehicleWithPhotoUri` (below) needs to build its multipart part. Everything else about `VehicleRepository.create()`'s write (local row, `OutboxWriter<T>` transaction) is unchanged from the earlier update.
4. **Uploading**: `packages/api-client` gains `createVehicleWithPhotoUri(client, payload, { uri, name, type })`, a sibling to the existing web-only `createVehicleWithPhoto(client, payload, photo: File)` sharing a `buildCreateVehicleFormData()` helper. They differ only in the type of the file part appended to `FormData`, because `File` is a DOM type with no React Native equivalent, while RN's own `FormData` implementation accepts a `{ uri, name, type }` descriptor that DOM's `FormData.append()` typing doesn't model — the one place a cross-platform package's shared types meet a platform-specific runtime shape. `outboxHandlers.ts`'s `CREATE_VEHICLE` handler calls `createVehicleWithPhotoUri` when the payload has a `photo`, `createVehicle` otherwise; retryable/permanent classification is unchanged.
5. **Cleanup**: the handler deletes the local stable-storage file after the upload either succeeds or is rejected permanently (4xx) — in both cases nothing will ever try that file again. It is deliberately *not* deleted after a retryable (5xx/network) failure, since the next flush attempt needs to find it in the same place.

**No local "pending photo" preview on Garage or Vehicle Detail.** Between save and the next successful sync, a newly-created Vehicle with a photo shows the same photo-less placeholder as one created without a photo — the real `photoUrl` appears once `SyncService.pull()`'s existing per-vehicle `applyDetail()` catches up, exactly like every other server-confirmed field already works today. Showing the local file in the meantime was considered and deliberately cut: it would need a new local-only column threaded through `VehicleSummary`'s mobile-local extension (`findAll()`'s return type, which `GarageScreen` renders directly), carried forward by `reconcile()` like the existing `VehicleDetailFields`, and cleared once the confirmed `photoUrl` arrives — real scope, and the Add Vehicle screen's own live preview (which does show the pick immediately, matching the web app's live preview column) already covers the moment where an Owner would actually look for it.

This does not change the outbox pattern, the outbox table schema, the conflict policy, or the client-generated-id mechanism above — it is one more concrete mechanism for a specific entry's payload, the same relationship the `OutboxWriter<T>` update and the client-supplied-`id` update above each have to this ADR's original Decision.

### Update (2026-07-03): two bugs found live-testing the photo upload update above, both fixed same-day

Live on-device testing (not covered by any of this feature's mocked unit tests) surfaced two real bugs the moment a photo was actually picked and synced. Both are corrections to the "offline-durable photo upload" update directly above, not new decisions.

**Bug 1 — `createVehicleWithPhotoUri`'s `{ uri, name, type }` descriptor doesn't actually work.** Every attempt failed with `Error: Unsupported FormDataPart implementation`. Cause: this app's global `fetch` is Expo's own "winter" runtime fetch (SDK 57), not React Native's built-in Networking-bridge `fetch` — and Expo's `FormData` conversion (`expo/src/winter/fetch/convertFormData.ts`) only accepts a string or a real `Blob`-like value (checked via a `.bytes()` method) for a file part. The classic RN `{ uri, name, type }` descriptor step 4 above described is a convention specific to RN's *other* fetch implementation, not this one — it was never going to work here. **Fix:** the outbox entry's payload still carries that plain descriptor (nothing else changes — it still needs to be JSON-serializable to survive in SQLite across an app restart), but `outboxHandlers.ts`'s `CREATE_VEHICLE` handler now opens a live `expo-file-system` `File` handle on the stable path (`photoStorage.ts`'s new `openVehiclePhotoFile()`) immediately before calling `createVehicleWithPhotoUri` — `File` `implements Blob`, satisfying Expo's fetch. `createVehicleWithPhotoUri`'s parameter type changed from the `{ uri, name, type }` descriptor to a minimal `UploadableFile` interface (`name`, `type`, `bytes()`) to match.

**Bug 2 — a concurrent sync could delete the photo file out from under a still-in-flight upload of the same file**, surfacing as `Error: ... couldn't be opened because there is no such file`, reproduced via a manual pull-to-refresh on Vehicle Detail shortly after creating the Vehicle. Cause: `SyncProvider.runFullSync()` has no reentrancy guard — mount, reconnect, app-foreground, and an explicit `refresh()` (pull-to-refresh) can all fire close together, and nothing stops two calls from running concurrently. `SyncService.flushOutbox()` has no locking either, so two overlapping calls can both pick up the same still-`pending` outbox entry. For a plain `CREATE_VEHICLE`/`UPDATE_VEHICLE` this race is harmless — both calls just make the same idempotent upsert/PATCH again. For a photo, it isn't: whichever call's upload finishes first deletes the local file (this update's own cleanup step, above); if the other call is still mid-upload reading that same file, it now finds nothing there. **Fix:** `SyncProvider.runFullSync()` is now single-flight, mirroring `TokenHttpClient.refreshOnce()`'s existing pattern in this same app — a `useRef<Promise<void> | null>` holds the in-flight sync; an overlapping call reuses it instead of starting a second `SyncService.runFullSync()`.

This also explains a third symptom seen in the same manual test: the newly-created Vehicle disappeared (Vehicle Detail showed "not found") after that same pull-to-refresh. `VehicleRepository.reconcile()` (2026-07-02 update, above) replaces the entire local Vehicles table with exactly what `GET /vehicles` returns — a Vehicle whose own `CREATE_VEHICLE` entry hasn't reached the server yet isn't in that list, so a pull *at that moment* removes it locally too. This is a **known, pre-existing gap**, not something this update fixes: it predates photo upload entirely (any Add Vehicle, photo or not, has a brief window between local write and confirmed sync where a pull would do this), and normally goes unnoticed only because `flushOutbox()` runs immediately before `pull()` in the same `runFullSync()` call, so the create has almost always already succeeded by the time reconcile runs. Bug 1 above defeated that ordering (the create kept failing, so it was never confirmed), which is what made this window wide enough to actually hit. Revisit as its own piece of work if it needs a real fix (e.g., `reconcile()` preserving rows with a still-pending `CREATE_VEHICLE` outbox entry) — not attempted here.

### Update (2026-07-03): local photo preview, and Garage's own missing focus-refetch

More live testing, after the two bugs directly above were fixed: the picked photo still didn't appear on Vehicle Detail right after creating the Vehicle, and returning to Garage from it didn't show the new Vehicle at all until an explicit pull-to-refresh. Both are corrections to this same day's work, not new architectural decisions.

**No local photo preview, revisited.** The "offline-durable photo upload" update above deliberately decided against a local preview (`Decisions` table in `docs/specs/mobile-app/vehicle.md`), reasoning that the Add Vehicle screen's own live picker preview already covered the moment an Owner would look for it. Asked directly, the answer was no — the photo should be visible on Vehicle Detail (and, for the same reason, Garage's card) immediately, not just once sync catches up. That reasoning is superseded.

**Decision:** no new local-only column. `VehicleRepository.create()` now stores the *stable local file path* (`persistVehiclePhoto()`'s return value, not the picker's own cache-scoped one) directly in the existing `photoUrl` field instead of `null`. `Image` renders a `file://` uri exactly like an `https://` one, so no UI change was needed on Garage or Vehicle Detail at all — both already just render `vehicle.photoUrl`. The next successful `reconcile()` (2026-07-02 update) spreads the incoming server row directly (`{ ...vehicle, ...carriedForwardDetailFields }`, where `photoUrl` comes from `vehicle`, not the carried-forward fields), so the moment the create+photo is confirmed, the real CDN url naturally overwrites the local path — no explicit "clear the pending photo" step needed. This works cleanly *because* the photo is bundled into the same atomic `CREATE_VEHICLE` request as the vehicle itself (this update's own earlier decision): there's no partial state where the vehicle syncs but the photo doesn't, so there's nothing in between "still fully local" and "fully confirmed, including the real photo" for `reconcile()` to get wrong.

A hypothetical second local-only column (mirroring `VehicleDetailFields`'s carry-forward-until-confirmed pattern) was the alternative considered and rejected — it would need `reconcile()` to know when to drop it (checking the incoming row's `photoUrl`), which the field-reuse approach gets for free from `reconcile()`'s existing spread order.

**Garage never got Vehicle Detail's `useFocusEffect` fix.** Vehicle Detail's own 2026-07-03 fix (see the note on `useVehicleDetailViewModel` in this file's history) already moved off a plain mount-`useEffect` for exactly this reason — native-stack doesn't remount a screen revealed via `router.back()`, so a Vehicle created while Garage sat inert on the stack (Add Vehicle → `replace()` → Detail → `back()` → the same Garage instance) never re-read local SQLite; only a completed sync (changing `lastSyncedAt`) or a fresh mount would trigger it. Garage's own data effect was never given the same fix when Add Vehicle was built, so this exact scenario reproduced there too, and even more visibly (an empty-looking Garage rather than a stale one). **Fix:** `useGarageViewModel`'s fetch effect is now `useFocusEffect` instead of `useEffect`, same `[vehicleRepository, lastSyncedAt]` dependency as before.

This does not change the outbox pattern, the outbox table schema, the conflict policy, or any mechanism from the two updates directly above — it corrects this same day's own scope call on local preview, and closes a gap in Garage that mirrors one Vehicle Detail had already fixed.

This does not change the outbox pattern, the outbox table schema, the conflict policy, or either mechanism from the "offline-durable photo upload" update directly above — both are bug fixes to that update's own mechanism, found the same day it was written.

### Update (2026-07-04): offline-durable photo upload extended to Edit Vehicle

This file's own "Out of scope" (in `docs/specs/mobile-app/vehicle.md`) had explicitly deferred editing an existing Vehicle's photo — "Add Vehicle only for now; needs its own step once those screens' photo affordances are designed." This update is that step, for Edit Vehicle (UC-MOB-VEH-6); Vehicle Detail's own photo affordance remains deferred.

**Decision:** reuse the exact mechanism the 2026-07-03 "offline-durable photo upload" update above already established for `CREATE_VEHICLE`, applied to `UPDATE_VEHICLE`:

1. **Picking/persisting**: identical to Add Vehicle — `expo-image-picker`, then `persistVehiclePhoto(vehicleId, photo)` before the write. No new mechanism needed here: `persistVehiclePhoto` was already keyed by the Vehicle's id rather than a fresh id, so Edit Vehicle (which always has a real, already-known `vehicleId`) calls it exactly the way Add Vehicle does.
2. **Local write**: `VehicleRepository.update()` gains an optional third parameter, `photo?: PickedPhoto`, mirroring `create()`'s. When present, the stable local path becomes the row's `photoUrl` (same reasoning as the 2026-07-03 "local photo preview" update — no second local-only column, `reconcile()` already knows how to overwrite `photoUrl` with the server's confirmed value); the `UPDATE_VEHICLE` outbox entry's payload gains the same optional `photo: { uri, name, type }` shape `CREATE_VEHICLE`'s payload already carries.
3. **Uploading**: `outboxHandlers.ts`'s `UPDATE_VEHICLE` handler awaits `updateVehicle(client, vehicleId, data)` (unchanged), then, only if the payload has a `photo`, opens a `File` handle (`openVehiclePhotoFile`, the same Bug-1 fix from 2026-07-03) and uploads it via a new `setVehiclePhotoUri(client, vehicleId, file)` — a sibling to `createVehicleWithPhotoUri`, hitting the same `POST /vehicles/:id/photo` endpoint the API already exposes (built for, and already unit-tested against, this exact request shape — nothing server-side changed here). A web-only `setVehiclePhoto(client, vehicleId, photo: File)` sibling exists for the same reason `createVehicleWithPhoto` does: `packages/api-client` is shared, and a DOM `File` has no React Native equivalent.
4. **Cleanup**: the local file is deleted once the *whole* entry reaches a terminal outcome — success, or a permanent (4xx) rejection of either the field `PATCH` or the photo `POST` — and kept for a retryable (5xx/network) failure of either, so the next flush attempt can still find it. This is a deliberate difference in shape from `CREATE_VEHICLE`'s handler, not a new decision: there, the create and the photo are one combined multipart request, so one call either fully succeeds or fully fails together. Here, the field `PATCH` and the photo `POST` are necessarily two separate requests — `PATCH /vehicles/:vehicleId` is JSON-only, and the multipart photo endpoint doesn't accept the other fields — so the handler's single `try/catch` around both calls is what keeps their success/failure and cleanup semantics equivalent to the single-request case: a permanent failure of *either* call ends the entry (and the file) the same way a permanent failure of `CREATE_VEHICLE`'s one call does.

**Why not a second outbox entry type for the photo, mirroring `CREATE_VEHICLE`'s decision:** the 2026-07-03 update's reasoning against a second `UPLOAD_VEHICLE_PHOTO` entry type (needing a local "photo still pending" field, plus ordering logic) applies identically here, with one difference that made the choice easier rather than harder: an `UPDATE_VEHICLE` entry has no ordering dependency on a prior entry the way a photo-for-a-not-yet-created-Vehicle would, since the Vehicle already exists server-side by the time an edit is possible at all. Bundling remains the simpler option regardless.

**No new local preview mechanism needed.** The 2026-07-03 "local photo preview" update's reasoning — reuse `photoUrl` directly, no second column — already generalizes to an existing Vehicle whose photo is being replaced: `VehicleRepository.update()` overwrites the row's `photoUrl` with the new stable local path the same way `create()` sets it on a fresh row, and the next `reconcile()` overwrites it again with the server's confirmed url once the upload lands, exactly as before.

This does not change the outbox pattern, the outbox table schema, the conflict policy, or the `CREATE_VEHICLE` mechanism from the two updates above — it is the same photo-upload mechanism, extended to a second outbox entry type that needed it.

### Update (2026-07-04): local detail cache (notes + items) for Log Entries, fetched via a new phase after reconcile

Gap found while starting Edit Log Entry (UC-MOB-LOG-2, UC-MOB-LOG-3): `log_entries`' local schema (this ADR's 2026-07-03 "child-collection pull" update) only carries `LogEntrySummary`'s fields — title, date, mileage, `itemCount`/`mediaCount` counts, `totalCost` — because that's all `GET /vehicles/:vehicleId` (the only pull source for Log Entries) returns. Neither `notes` nor the actual item rows (`categoryId`/`description`/`quantity`/`unitCost`) are in that response at all; the only endpoint that returns them is `GET /vehicles/:vehicleId/log/:entryId` (`LogEntryDetail`), which nothing in the pull path has ever called. Edit needs both to pre-fill the form, and since `PATCH` does a full-array replace of items (per `log-entry-api.md`), silently editing with an empty cached item list would delete the Owner's real items on save — a correctness risk, not just a missing-field inconvenience.

Two approaches were considered (put to the user directly, since this is a genuine offline-architecture fork, not a mechanical choice):

- **Fetch on-demand when Edit Log Entry opens** — a new targeted `SyncService` method the viewmodel triggers if detail isn't cached yet. Avoids extending `pull()`, but is a new exception to "viewmodels never call services" (this ADR's and `apps/mobile/CLAUDE.md`'s existing rule), and means an entry can't be edited offline until its detail has been fetched at least once online.
- **Extend `pull()` to fetch full detail, same shape as the 2026-07-03 Vehicle Detail update** — add a phase that calls `GET /vehicles/:vehicleId/log/:entryId` per entry and caches `notes`/items locally via a `LogEntryRepository.applyDetail()` sibling to `VehicleRepository.applyDetail()`. Edit keeps reading purely from SQLite, no new viewmodel→API path. Cost: one extra sequential GET per Log Entry per pull — chosen to run **only for entries not already known locally with detail**, not every entry every pull (unlike Vehicle Detail's per-vehicle refetch, which is fine at "a handful of vehicles" scale but not at "dozens to low hundreds of Log Entries" scale, per `docs/adr/0018-log-entry-data-model.md`'s own sizing). Once an entry's detail has been fetched once (or it was created/edited on this device, which already has the full data in hand), it is never re-fetched by a later pull — this is a deliberate scope-narrowing versus Vehicle Detail's own precedent, accepted because Log Entries are effectively append-only from the mobile client's point of view (edits happen through this same device's `update()`, which already writes the authoritative local copy).

**Decision:** the second option (extend `pull()`), scoped to first-seen entries only.

**Local schema** (`log_entries`): three new columns — `notes TEXT`, `items_json TEXT NOT NULL` (JSON-serialized `Array<{ categoryId, description, quantity, unitCost }>`, mirroring `LogEntryRepository.CreateLogEntryItemData`), `detail_fetched INTEGER NOT NULL DEFAULT 0`. `items_json` (not a second table) because `Store<T>` is a single-table, generic port with no join support, and a Log Entry's item count is small — a JSON blob column is the same "don't build a bigger primitive than the one write path needs" reasoning this ADR's `OutboxWriter<T>` update already applied. `detail_fetched` is a real column, not `notes`/`items` nullability alone, because `notes: null` and `items: []` are legitimate *known* states (a minimal Log Entry genuinely has neither) — an explicit flag is the only way to distinguish "confirmed empty" from "not yet fetched."

**`LogEntryRepository`** gains:
- `findById(id): Promise<LogEntryFullDetail | null>` — reads the single local row, parses `items_json`. The read path Edit Log Entry's viewmodel uses; still purely local, no change to the "all reads come from SQLite" rule.
- `update(vehicleId, entryId, data)` / `delete(vehicleId, entryId)` — `OutboxWriter<T>`-backed, same shape as `create()`/`VehicleRepository.update`/`.delete`, enqueuing `UPDATE_LOG_ENTRY`/`DELETE_LOG_ENTRY`. `update()` writes `notes`/`items_json`/`detail_fetched: true` directly from the data the Owner just edited — no round trip needed to know its own just-written detail.
- `applyDetail(id, { notes, items })` — `VehicleRepository.applyDetail`'s sibling; merges fetched detail into an existing row, sets `detail_fetched: true`.
- `reconcile(entries)` now returns `Promise<string[]>` (the ids needing a detail fetch) instead of `Promise<void>`: it looks up each incoming entry's existing local row before replacing the table (same carry-forward shape as `VehicleRepository.reconcile`'s detail fields), keeps `notes`/`items_json`/`detail_fetched` for any row already `detail_fetched: true`, and defaults the rest to `null`/`'[]'`/`false` while collecting their ids to return.

**`SyncService.pull()`** gets a third phase after the existing per-vehicle detail loop and the single `logEntryRepository.reconcile(allLogEntries)` call: for each id `reconcile()` returned, call `getLogEntry(client, vehicleId, entryId)` and `logEntryRepository.applyDetail(entryId, { notes, items })`. A failed per-entry fetch (network blip, entry deleted server-side between phases) is logged and skipped, same "stays stale until next pull, never a dangling reference" tradeoff this ADR already accepts everywhere else — and since `detail_fetched` stays `false`, the next pull's `reconcile()` will list it again, so a transient failure self-heals instead of permanently losing the entry's detail.

This does not change the outbox pattern, the outbox table schema, the conflict policy, or any prior update's mechanism — it is a new local-cache column set and a new (bounded) pull phase, answering a question ("how does Edit Log Entry get notes/items") none of this ADR's Log Entry work had needed to answer before now.

## Status

accepted

## Consequences

- Writes are instant from the user's perspective — no spinner waiting for the network on create/update/delete.
- The outbox is a plain SQLite table; no third-party sync library or hosted service is required.
- Idempotency keys prevent duplicate writes on retry; the server-side implementation of idempotency keys is additive (no breaking API changes).
- The pull-on-refresh step means an Owner who opens the app after a long offline period gets a full reconcile, which is correct.
- Failed outbox entries are surfaced to the SyncService for monitoring; V2 can add user-facing retry or conflict resolution UI.

## V2+ items

- **Incremental pull** — once data volumes grow, replace the pull-everything reconcile with a cursor-based pull. Requires soft deletes on the API and a `changedSince` cursor endpoint.
- **User-facing conflict resolution** — when the server rejects a local mutation, surface the conflict to the Owner rather than silently reverting.
- **Outbox retry backoff** — exponential backoff on `failed` entries to avoid hammering a degraded API.
